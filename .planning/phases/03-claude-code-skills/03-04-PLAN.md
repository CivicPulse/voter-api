    Instructions (50+ lines):

    1. **Input** section:
       - Accept PDF path as $0 (single PDF or directory containing PDFs)
       - Accept election date as $1 (YYYY-MM-DD format)
       - Read the PDF natively (Claude reads PDFs directly -- no parsing library)

    2. **Format Reference** section:
       - Read: @docs/formats/markdown/election-overview.md
       - Focus on the Calendar section with its metadata fields

    3. **Extraction Fields** section (from CONTEXT.md):
       Core dates:
       - Election Day (from PDF or $1 argument)
       - Registration Deadline
       - Early Voting Start
       - Early Voting End
       - Absentee Ballot Application Deadline

       Absentee dates:
       - Earliest Day to Mail Absentee Ballot
       - Earliest Day Voter Can Request Mail Ballot

       From master calendar:
       - Qualifying Period Start
       - Qualifying Period End

       Validation:
       - Participating Counties List (cross-check against counties/ directory)

    4. **PDF Types** section:
       - Master calendar PDF ("2026 Short Calendar .pdf"): Contains full election cycle view. Extract qualifying period dates.
       - Per-election PDF ("MAY_19_2026-...DATA.pdf"): Contains specific dates for one election. Extract all core and absentee dates.
       - If directory provided: process all PDFs found, correlating dates to the target election

    5. **Output** section:
       - Update the existing election overview file at data/elections/{date}/{date}-{type}.md
       - Add/update the Calendar section with extracted dates in the metadata table
       - All dates in YYYY-MM-DD format for metadata fields
       - Show diff before writing, ask for confirmation (unless --direct)

    6. **Error Handling** section:
       - PDF unreadable -> report error, skip
       - Date not found in PDF -> leave field empty, warn
       - Multiple elections in one PDF -> ask user which to extract

    Create .claude/skills/candidate-enrichment/SKILL.md:

    Frontmatter:
    ```yaml
    ---
    name: candidate-enrichment
    description: Enrich candidate markdown files with bios, photo URLs, and contact info from web research
    argument-hint: <candidates-dir> [--depth full|basic|minimal] [--direct]
    disable-model-invocation: true
    allowed-tools: Read, Write, Grep, Glob, Bash, WebFetch
    ---
    ```

    Instructions (80+ lines):

    1. **Input** section:
       - Accept candidates directory as $0 (typically data/candidates/)
       - Parse --depth flag from $ARGUMENTS: full (default), basic, minimal
       - Parse --direct flag: skip interactive confirmation

    2. **Format Reference** section:
       - Read: @docs/formats/markdown/candidate-file.md
       - Read: @.claude/skills/includes/format-rules.md

    3. **Depth Levels** section:
       - minimal: Name verification only (confirm identity, no enrichment)
       - basic: Bio, party affiliation, current office (if incumbent)
       - full: Bio, photo URL, campaign website, social media, contact info, education, career

    4. **Research Sources** section (priority order):
       a. Ballotpedia (primary): Structured profiles with bio, photo, positions
          - Search: "ballotpedia.org {candidate name} {state} {office}"
          - Extract: biography, photo URL, prior offices, education
       b. Campaign websites: Bio, platform, photo, contact
          - Use URL from candidate file if present, validate it's live
          - Extract: bio paragraph, photo URL, issues/platform
       c. Official government sites: GA SOS, state legislature, county sites
          - For incumbents: official portrait, committee assignments
       d. Social media: LinkedIn, Twitter/X, Facebook
          - Extract: profile photo, headline/bio, links

    5. **Confidence Annotations** section:
       - All enriched data MUST include source annotation
       - Format: `(source: ballotpedia)`, `(source: campaign-website)`, `(unverified)`
       - Use `(unverified)` when source is ambiguous or candidate identity uncertain
       - Normalizer will strip annotations before converter runs
       - Example: `Photo URL: https://example.com/photo.jpg (source: ballotpedia)`

    6. **Processing Steps** section:
       Step 1: Scan candidates directory, list all .md files
       Step 2: For each candidate file, read existing content
       Step 3: Identify what fields are missing/empty (bio, photo, contact, etc.)
       Step 4: Research using sources in priority order
       Step 5: Update candidate file with enriched data + source annotations
       Step 6: Show diff and ask for confirmation (unless --direct)
       Step 7: Validate URLs are live (check response status)
       Step 8: Report: X candidates enriched, Y fields added, Z URLs validated

    7. **Deduplication** section:
       - Same-name candidates in different contests: verify identity via web search
       - If uncertain: mark with (unverified) and flag for human review
       - Never merge candidates without confirmation

    8. **URL Validation** section:
       - For every URL added: check that it responds (use WebFetch or curl)
       - Dead links: mark with `(dead link as of YYYY-MM-DD)` annotation
       - Redirected URLs: follow redirect, use final URL

    9. **Batch Processing** section:
       - Process candidates sequentially by default
       - Report progress: "Enriching candidate X of Y: {name}"
       - If a candidate fails enrichment: log warning, continue to next
       - Checkpoint: after each candidate, the file is saved (resume-friendly)

    10. **Agent Team Coordination** section (when Claude Code cooperative team is available):
        - Coordinator pre-scans all candidate files, counts fields to enrich
        - Partition candidates into balanced batches (aim for 4 workers)
        - Each worker processes its batch independently
        - Workers report results back to coordinator
        - Coordinator resolves any dedup conflicts

    Create .claude/skills/process-election/SKILL.md:

    Frontmatter:
    ```yaml
    ---
    name: process-election
    description: Full election data pipeline - process SOS CSV through skills, normalize, and prepare for review
    argument-hint: <csv-file-path> [--direct] [--skip-enrich]
    disable-model-invocation: true
    ---
    ```

    Instructions (40+ lines):

    1. **Pipeline Overview** section:
       This skill orchestrates the full election data processing pipeline:
       ```
       SOS CSV -> qualified-candidates skill -> normalize elections -> normalize candidates -> [optional: enrich] -> review via git diff
       ```

    2. **Input** section:
       - Accept CSV path as $0
       - Parse --direct flag (pass through to sub-skills)
       - Parse --skip-enrich flag (skip candidate enrichment step)

    3. **Pipeline Steps** section:

       Step 1: **Validate Input**
       - Verify CSV file exists and is readable
       - Determine election date from filename (e.g., may-19-2026-qualified-candidates.csv -> 2026-05-19)

       Step 2: **Process Qualified Candidates**
       - Run the qualified-candidates skill on the CSV
       - Pass --direct flag if provided
       - Verify output directory created: data/elections/{date}/
       - Verify candidate stubs created: data/candidates/

       Step 3: **Normalize Election Files**
       - Run: `uv run voter-api normalize elections data/elections/{date}/`
       - Review normalizer report for warnings
       - If errors: stop pipeline, report issues

       Step 4: **Normalize Candidate Files**
       - Run: `uv run voter-api normalize candidates data/candidates/`
       - This generates UUIDs and renames placeholder files
       - Review normalizer report for warnings

       Step 5: **Enrich Candidates** (unless --skip-enrich)
       - Run candidate-enrichment skill on data/candidates/ with --depth basic
       - This is the longest step -- provides progress updates

       Step 6: **Review**
       - Run `git diff --stat` to show all changes
       - Summarize: X election files created, Y candidate files created, Z candidates enriched
       - Prompt user to review changes before committing

    4. **Error Recovery** section:
       - Each step is independent -- if step 3 fails, steps 1-2 output is preserved
       - User can re-run pipeline with same CSV (idempotent within same session)
       - Pipeline reports which step failed and how to resume manually

    5. **Election Calendar Integration** section:
       - If SOS calendar PDFs are available in data/import/, suggest running /election:calendar after pipeline
       - Calendar processing is NOT part of the main pipeline (separate manual step)
