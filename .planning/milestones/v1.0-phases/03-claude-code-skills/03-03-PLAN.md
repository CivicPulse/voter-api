    4. **Contest Name Parsing** section:
       - Read: @.claude/skills/includes/contest-patterns.md
       - This is the core AI task: parse CONTEST NAME into body_id, seat_id, election_type
       - NEVER use deterministic rules for this -- use AI pattern matching informed by the examples
       - Handle party suffixes: strip "(R)", "(D)", "(L)", "(I)" from contest name, record as metadata

    5. **Diff-Aware Updates** section (per user decision on diff-aware updates):
       - Before generating output, check if the target election directory already exists: `data/elections/{date}/`
       - If directory exists:
         a. Report: "Election directory already exists with N files."
         b. Ask operator: "Regenerate from scratch (overwrites all files) or Update mode (only add/modify changed candidates, preserve manual edits and enrichment)?"
         c. If --direct flag is set, default to "regenerate" mode
       - **Regenerate mode**: Delete existing directory contents and regenerate all files fresh from CSV
       - **Update mode**:
         a. Load existing markdown files, extract current candidate lists per contest
         b. Compare CSV candidates against existing candidates
         c. Add new candidates not in existing files
         d. Update changed fields (status, party, qualified date) for existing candidates
         e. Preserve fields not in CSV (bio, photo, enrichment data, manual edits)
         f. Report: "Added X new candidates, updated Y fields, Z files unchanged"
       - If directory does not exist: proceed normally (fresh generation)

    6. **Checkpoint & Resume** section (per dual checkpointing decision in CONTEXT.md):
       Skills run in Claude Code context (not as API requests), so use JSONL-only checkpointing:

       **Checkpoint file location**: `data/elections/{date}/.checkpoint.jsonl`
       (Gitignored via the data/import/.gitignore pattern -- add a note to also gitignore .checkpoint.jsonl files in the election directory)

       **Writing checkpoints**: After each contest is fully processed (contest file written + candidate stubs created), append a JSONL line:
       ```json
       {"contest_name": "Board of Education District 3", "file_written": "2026-05-19-fulton-boe-district-3.md", "candidates_created": 4, "timestamp": "2026-03-15T10:30:00Z"}
       ```

       **Detecting checkpoints on re-invocation**: Before processing begins:
       a. Check if `data/elections/{date}/.checkpoint.jsonl` exists
       b. If it exists, read all lines and build a set of already-processed contest names
       c. Report: "Found checkpoint with N contests already processed. Resuming from contest N+1."
       d. Skip contests already in the checkpoint set
       e. Continue processing remaining contests normally, appending new checkpoint lines

       **Checkpoint cleanup**: When all contests are processed successfully, delete the checkpoint file. A completed run needs no checkpoint.

       **Checkpoint + regenerate mode interaction**: If the user chooses "regenerate" mode and a checkpoint exists, delete the checkpoint file first (starting fresh).

    7. **Processing Steps** section:
       Step 1: Determine election date from CSV filename or first row context
       Step 2: Group rows by CONTEST NAME (after stripping party suffix)
       Step 3: Check for checkpoint file and load already-processed contests (see Checkpoint & Resume section)
       Step 4: For each contest group (skipping checkpointed ones), infer body_id, seat_id, election_type from the contest name
       Step 5: Classify contests: statewide/federal (COUNTY varies), county-specific (single COUNTY), municipal (MUNICIPALITY non-empty)
       Step 6: Deduplicate candidates -- same (CANDIDATE NAME, POLITICAL PARTY) appearing in multiple county rows = one candidate
       Step 7: Create election directory: data/elections/{date}/
       Step 8: Create overview file: {date}-{election-type}.md with Metadata table (ID empty, Format Version 1, election_type, stage)
       Step 9: For statewide/federal contests: create single-contest files in election directory root
       Step 10: For county contests: group by county, create multi-contest files in counties/ subdirectory
       Step 11: For each unique candidate: create stub file in data/candidates/ with placeholder filename ({name-slug}-00000000.md)
       Step 12: After each contest: write checkpoint line (see Checkpoint & Resume section)
       Step 13: After all contests: delete checkpoint file

    8. **Output Structure** section:
       ```
       data/elections/{date}/
       |- {date}-{type}.md           # Overview
       |- {date}-{contest-slug}.md   # Statewide contests
       |- .checkpoint.jsonl          # Temporary, deleted on completion
       \- counties/
           \- {date}-{county}.md     # County files
       data/candidates/
       \- {name-slug}-00000000.md    # Candidate stubs
       ```

    9. **Interactive Mode** section:
       - Default: show each file's content and ask for approval before writing
       - If --direct flag: write directly, review via git diff
       - Show progress: "Processing contest X of Y..."

    10. **Error Handling** section:
       - Missing/renamed CSV columns -> immediate failure with expected vs found
       - Unparseable contest name -> log warning, skip contest, continue
       - Duplicate candidate name collision (same slug) -> append sequence number to placeholder filename
       - Interrupted processing -> checkpoint file preserves progress, re-invocation resumes

    11. **Quality Checks** section:
       - After generation: count files created, candidates created
       - Compare candidate count against unique (CANDIDATE NAME, POLITICAL PARTY) pairs in CSV
       - If mismatch: warn about potential dedup issues
