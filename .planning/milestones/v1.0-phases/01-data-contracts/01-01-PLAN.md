Do NOT delete the old files -- they remain for reference until Phase 2 migration.

Use the existing format specs in `data/elections/formats/` as content basis -- preserve the content rules (title case, URL normalization, occupation formatting, em-dash for empty values, etc.) and extend with new fields.

For ALL five formats, the metadata table MUST include:
- `| ID | {uuid} |` row (required -- converter error if missing)
- `| Format Version | 1 |` row (required -- starting at version 1)

1. **election-overview.md** — Unified overview format (replaces two variants: General/Primary and Special Election).
   - H1: `{Month D, YYYY} -- {Election Display Name}`
   - Metadata table with: ID, Format Version, Name (SOS) (exact SOS string), Date, Type (election_type value from vocabulary), Stage (election_stage value, default: election), Body (body ID), Seat (seat ID)
   - For overview files (ElectionEvent level): Body and Seat are typically not applicable (the event groups multiple contests). Include a note that these are per-event metadata -- district-level Body/Seat lives on individual contest files.
   - Calendar dates section with: Registration Deadline, Early Voting Start, Early Voting End, Absentee Request Deadline, Qualifying Period Start, Qualifying Period End, Election Day -- dates in ISO 8601 (YYYY-MM-DD) with Source column
   - Data Sources section with SOS feed URL (TBD until available), CSV sources, PDF sources
   - Optional sections: Statewide Races, Federal Races, State Legislative Races, Local Elections, Candidate Breakdown -- included based on what contests exist (not based on election type)
   - Validation checklist at the bottom

2. **single-contest.md** — Single-contest format (replaces STATEWIDE-CONTEST-FORMAT.md and SPECIAL-ELECTION-CONTEST-FORMAT.md).
   - H1: `{Contest Display Name}` (for statewide/federal) or `{Locality} -- {Office Title}` (for special elections)
   - Metadata table with: ID, Format Version, Election (link to overview), Type (election_type), Stage (election_stage), Body (body ID from vocabulary), Seat (seat ID from vocabulary), Name (SOS) (exact SOS string)
   - Boundary_type and district_identifier are resolved by the converter from Body/Seat via county reference files -- they do NOT appear in the markdown
   - Candidate table columns: Candidate (linked to candidate file), Status, Incumbent, Occupation, Qualified Date -- Email and Website columns are DROPPED (moved to candidate file per CONTEXT)
   - Internal structure adapts by election type: partisan primaries use party sections (`## Republican Primary`, `## Democrat Primary`) with `**Contest Name (SOS):**` lines; non-partisan/general/jungle use single `## Candidates` section
   - Party row in metadata is optional -- present only for partisan primaries
   - Preserve all content rules from existing statewide and special election formats (title case, em-dash, URL rules, occupation rules)
   - For special elections: include Purpose and Eligibility metadata fields, Candidate Bios section
   - Validation checklist

3. **multi-contest.md** — Multi-contest county format (replaces COUNTY-FORMAT.md).
   - H1: `{County Name} County -- Local Elections`
   - File-level metadata table with: ID, Format Version, Election (link to overview), Type, Contests count, Candidates count
   - Per-contest structure: `### {Contest Name in Title Case}` heading, followed by `**Body:** {body-id} | **Seat:** {seat-id}` on its own line, then candidate table
   - Candidate table columns: Candidate (linked to candidate file), Status, Incumbent, Occupation, Qualified Date -- Email and Website DROPPED
   - Preserve all content rules from existing county format (title case contest names, occupation formatting, em-dash)
   - Statewide & District Races cross-reference section (linking to statewide contest files)
   - Validation checklist

4. **candidate-file.md** — Global candidate file format (new, no predecessor).
   - Location: `data/candidates/{name-slug}-{8-char-hash}.md`
   - H1: `{Full Name}`
   - Person metadata table: ID, Format Version, Name, Photo URL, Email
   - `## External IDs` table: Source, ID columns (Ballotpedia, Open States, VPAP, etc.)
   - `## Links` table: Type (from link-types vocabulary), URL, Label columns
   - `## Elections` section with election-keyed subsections:
     - `### {Election Display Name} -- {Contest Name}`
     - Per-election table: Election ID, Contest File (relative link), Party, Occupation, Filing Status, Qualified Date, Is Incumbent
   - Document the 8-char hash in filename (first 8 chars of the UUID, for human disambiguation of common names)
   - Validation checklist

5. **county-reference.md** — County reference file format (extends existing stub format).
   - Location: `data/states/GA/counties/{county}.md`
   - H1: `{County Name} County`
   - County metadata table: County, State (link), County Seat, Consolidated Government (if applicable), Official Website
   - `## Governing Bodies` table: Body ID, Name, boundary_type, Seat Pattern, Notes
   - The Seat Pattern column documents the seat naming convention for that body (e.g., "at-large (posts 7-8), district-N (1-6)")
   - Note that the converter resolves Body IDs to boundary_type via this table -- if the Body ID from a contest file isn't found here, the converter emits a validation error (strict per CONTEXT)
   - Validation checklist
