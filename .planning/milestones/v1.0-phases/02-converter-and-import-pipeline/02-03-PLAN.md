    Election Event Import (test_election_event_import.py):
    - Reads JSONL file, validates each line against ElectionEventJSONL via model_validate()
    - Valid records -> bulk UPSERT into election_events table by UUID primary key
    - Re-import same file -> 0 inserts, N updates (idempotent, same data = no-op updates)
    - Dry-run mode -> returns {would_insert: N, would_update: M} without writing
    - Invalid JSONL line -> error logged, line skipped, import continues (unless --fail-fast)
    - Returns summary: {inserted: N, updated: M, skipped: K, errors: [...]}

    Election Import (test_election_import.py):
    - Same pattern as election event import but targets elections table
    - Validates against ElectionJSONL
    - election_event_id FK must reference existing election_event (or import fails that record)
    - Idempotent on UUID, dry-run supported

    Candidacy Import (test_candidacy_import.py):
    - Validates against CandidacyJSONL
    - candidate_id FK must reference existing candidate
    - election_id FK must reference existing election
    - Idempotent on UUID or (candidate_id, election_id) unique constraint
    - Dry-run supported

    Pipeline command:
    - `voter-api import election-data <dir>` runs imports in order: election-events.jsonl -> elections.jsonl -> candidates.jsonl -> candidacies.jsonl
    - Stops on first failure (FK dependency chain)
    - Reports what succeeded and what failed

    **Add migration command** to `src/voter_api/cli/convert_cmd.py`:
    - `@convert_app.command("migrate-format")` -- `voter-api convert migrate-format <dir> [--dry-run]`
    - Implements the rules from `docs/formats/specs/migration-rules.md` (read this file for complete rules)
    - For each file type:
      - **Overview files**: Add ID (empty), Format Version, Type, Stage metadata rows. Convert calendar table to ISO format with Source column. Add Data Sources section.
      - **Single-contest files**: Add ID (empty), Format Version, Body/Seat metadata line. Reduce candidate table from 7 to 5 columns (drop Email, Website). Add missing metadata rows (Type, Stage).
      - **Multi-contest files**: Add ID (empty), Format Version. Add Body/Seat metadata lines to each ### contest section. Reduce candidate tables from 7 to 5 columns.
    - Files are rewritten in-place
    - Files that fail validation are skipped with errors in the report
    - --dry-run mode: report what would change without writing
    - Migration must be idempotent: already-migrated files (detected by `Format Version` row presence) are skipped
    - Use defensive parsing: handle both 5 and 7 column tables, both ISO and "Month D, YYYY" date formats, em-dash vs hyphen
    - Print summary: N files migrated, M skipped (already migrated), K failed

    **Add UUID backfill command** to `src/voter_api/cli/convert_cmd.py`:
    - `@convert_app.command("backfill-uuids")` -- `voter-api convert backfill-uuids <dir> [--dry-run]`
    - This command DOES need DB access (matches files to existing DB records)
    - Uses asyncio.run pattern with init_engine/dispose_engine
    - For each election markdown file:
      1. Read metadata (file type, election name, date)
      2. Query DB for matching record by natural key (per backfill-rules.md):
         - ElectionEvent: match on (event_date, event_type)
         - Election: match on (name, election_date)
         - Candidate: match on (full_name) -- person entity
      3. If DB match found: write DB UUID into the file's `| ID | {uuid} |` row
      4. If no DB match: generate new UUID v4 and write it
      5. For multi-contest files: each ### contest section gets its own Election UUID (matched or generated)
    - Print summary: N UUIDs backfilled from DB, M new UUIDs generated, K files skipped/failed

    **Update E2E smoke tests** (`tests/e2e/test_smoke.py`):
    - Add test class `TestImportPipeline` with tests for:

      - `test_import_election_events_admin_list`: Admin can list import jobs (if existing endpoint handles it)
      - `test_import_cli_help`: Verify `voter-api import election-events --help` exits 0 (CLI registration test)
    - Update any existing import-related tests that may break due to new commands
    - Ensure existing `TestImports` tests still pass
