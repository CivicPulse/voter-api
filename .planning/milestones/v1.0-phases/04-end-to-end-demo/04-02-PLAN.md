    1. **Introduction**
       - What this walkthrough proves: the three-stage pipeline (SOS -> Markdown -> JSONL -> Database)
       - What you'll need: uv, Docker, jq, git, curl
       - Time estimate: ~15 minutes

    2. **Prerequisites**
       - Clone the repo, install dependencies (`uv sync`)
       - Reference `specs/001-voter-data-management/quickstart.md` for detailed setup
       - PostGIS via docker-compose (`docker compose up -d db`)
       - Apply migrations (`uv run voter-api db upgrade`)

    3. **Step 1: Review the Markdown (Human-in-the-Loop)**
       - Explain what Phase 3 produced: markdown files from SOS CSV data
       - Show `git log --oneline` for the Phase 3 commits that created the files
       - Show `git diff` or file listing to demonstrate the review checkpoint
       - Explain the design philosophy: AI produces data, humans review, deterministic tools import

    4. **Step 2: Prepare Files for Conversion**
       - Explain why May 19 needs format migration (pre-enhanced format)
       - Run and show: `voter-api convert migrate-format data/elections/2026-05-19/`
       - Explain March 10/17 already have enhanced format but empty UUIDs
       - Run and show: `voter-api normalize elections` for all three
       - Show before/after of a file's metadata table (ID populated)

    5. **Step 3: Convert Markdown to JSONL**
       - Explain the converter is deterministic (no AI) -- same input always produces same output
       - Run and show: `voter-api convert directory` for each election, using `--output` for per-election JSONL
       - **Show the full validation report** with annotations explaining each section (successes, warnings, failures)
       - Explain the file output structure (election_events.jsonl + elections.jsonl)
       - Note: the converter produces election events and election contests; candidate data is handled separately in the next step

    6. **Step 4: Generate Candidate and Candidacy JSONL**
       - Explain that the 49 `data/candidates/*.md` files contain enriched candidate data (bio, photo, contact info, external IDs) linked to specific election contests via Contest File references
       - Run and show: `uv run python scripts/generate_candidate_jsonl.py`
       - Show the summary output (N candidates, N candidacies per election date)
       - Explain how the script resolves election_id by reading the contest file's UUID
       - Show a sample `candidates.jsonl` record to demonstrate the enriched fields
       - Show a sample `candidacies.jsonl` record to demonstrate the candidate-election linkage

    7. **Step 5: Dry-Run Import**
       - Explain dry-run purpose: validates JSONL against DB without writing
       - Run and show: `voter-api import election-data <dir> --dry-run` for each
       - Show the "would insert N" output -- now includes counts for candidates and candidacies alongside election events and elections

    8. **Step 6: Import into Database**
       - Run and show: `voter-api import election-data <dir>` for each election
       - Show the pipeline summary (inserted/updated counts for all four JSONL types)
       - Document expected record counts per election

    9. **Step 7: Verify Idempotency**
       - Re-run the import commands
       - Show "0 inserted, N updated" output
       - Explain UPSERT behavior

    10. **Step 8: Verify via API**
        - Start the dev server (`uv run voter-api serve --reload`)
        - Create admin user: `voter-api user create --username demo --email demo@example.com --password "DemoP@ss123!" --role admin --if-not-exists`
        - Login and get JWT: `curl POST /auth/login` with JSON body (NOT form-data -- Phase 009 change)
        - Show all four query types with real curl commands and real response output (per CONTEXT.md locked decisions):
          a. **List elections by date**: `GET /api/v1/elections?date_from=2026-05-19&date_to=2026-05-19` -- verify count matches expected
          b. **Candidate lookup**: `GET /api/v1/candidates/{id}` -- verify enriched fields (bio, photo, contact info) came through
          c. **Election detail with candidates**: `GET /api/v1/elections/{id}` -- show contest with linked candidates via candidacy junction
          d. **District-based query**: query by boundary_type + district_identifier, prove Body/Seat district linkage survives the full pipeline
        - Include count assertions: "You should see N elections for May 19" etc.
        - Spot-check a specific candidate by name (e.g., a candidate from the Bibb Commission District 5 race) to verify data accuracy including bio and links

    11. **Cleanup**
        - Two options: keep database running for development, or `docker compose down`
        - Stop the dev server

    12. **What This Proved**
        - Summarize what the pipeline demonstrated
        - List what works (election events, elections, candidates, candidacies, district linkage, idempotency, enriched candidate fields)
        - Reference the architecture for future enhancements

    **Formatting rules (per user decisions):**
    - Use curl for API verification (per Claude's discretion -- zero dependencies, universal)
    - Use `jq` for JSON formatting in curl examples
    - Include real terminal output in fenced code blocks
    - Target audience: developer with uv + PostGIS, familiar with CLI + git
    - Happy path only -- no error scenarios
    - All env vars shown explicitly (DATABASE_URL, JWT_SECRET_KEY, ELECTION_REFRESH_ENABLED)

    **To capture real output:** If the 04-01-SUMMARY.md contains the terminal output, use that directly. If not, re-run the pipeline commands against the local PostGIS database to capture fresh output. The database should still have the imported data from Plan 01.
